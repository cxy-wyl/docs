  1. 方案1：响应拦截器判断401，弹框弹出后全局变量为true，后续再有401，如果变量为true就不弹，设定时器将变量变为false

问题：如果设3s后变为fasle，这时已经转到登录页重新登录成功了，有个旧的请求超时5s才返回就又转登录页反复跳转过程。个别请求返回慢会造成数据紊乱，不只是这个页面，还有列表详情页面等点进去又关闭会出现这种情况。token失效请求超时很少见一般很快，但是列表到详情页a然后跳转到b,b渲染会受a请求的数据影响这种情况可能出现。

  1. 方案2：检测到当前路由变化也就是跳转登录页时，结束当前页面的所有promise请求。检测路由变化后取消当前页面的promise请求 。

RBAC方案 ​

  • 传统：用户与权限多对多，每次新增用户，依次分配权限
  • 现在：新增用户，分配角色，角色与权限多对多，每个角色对不同权限
  • 权限分为： 数据权限(后端)、路由权限、按钮权限，与后端配合从接口权限，按钮 disable 状态控制，菜单权限，路由守卫拦截页面跳转几方面对用户访问权限进行控制
  • 数据权限：
    • 用户访问接口
    • 判断用户和当前path，根据用户->角色->页面/功能->接口 得到当前用户有权限的接口列表与当前path相比
    • 若无权限（某些接口只需要登录就能访问的，比如获取用户姓名信息的需要排除在外）则直接返回失败，前端全局捕获后给出无权限提示
  • 按钮权限
    • 客户端登录后请求用户对应的权限列表(路由 按钮)v-hasRole   v-hasPermi
  • 路由权限：要考虑扩展性 安全性 灵活性
    • 方案1：纯前端
      • 问题：灵活主要对角色，不能保证A没有B权限url输入b相关的路由也能访问B的路由
      • 过程：路由钩子，有token进行权限校验，没有token未登录。有token且访问login页面，重定向到首页，登录重定向，已经登陆过。有token且访问其他路由页面，发请求拿用户信息里面包含角色，后端只给角色，前端根据角色动态生成路由，如果没有当前要访问路由的权限就导航到401，否则继续访问
    • 方案2：后端掌控
      • 问题：安全性高，死板要配置，前端只写公共router。ui界面操作为路由新增一条数据，立即更改动态生效。如果权限本身可配，推荐，不用改代码
      • 过程： 路由新增页面时候就确定，给到后端，路由由后端返回携带用户相关路由path信息，前端根据路由信息拼凑再动态生当前用户的路由。后端没有返回额外的不属于当前用户的路由信息，前端会跳转到404。但是，可以优化：前端访问找不到返回404，404代表没资源，但当前是有资源但没权限，所以返回403，前端有所有的路由表去比对决定返回403还是404。
      • 对应的功能：菜单管理：配路由地址，路由权限字符，按钮权限字符角色管理：勾选该角色对应的菜单及按钮，给角色添加权限字符部门管理：数据控制，下级看不到上级用户管理：新增-分配角色-分配部门

按钮权限 v-hasPermi

js
//给v-hasPermi传入数组包含权限字符串，有权限字符串的就可以显示操作按钮
export default {
    inserted(el, binding, vnode) {
        // 主页面绑定的value
        const { value } = binding;
        const all_permission = "*:*:*";
        // 取出当前user的权限
        const permissions = store.getters && store.getters.permissions;
        // 绑定的value必须是数组
        if (value && value instanceof Array && value.length > 0) {
            const permissionFlag = value;
            // 遍历当前user的权限列表，新增时每个按钮有一个权限字符，一个true就返回true,有权限不进行任何操作
            const hasPermissions = permissions.some((permission) => {
                // 如果是admin就是"*:*:*"
                // 或者看业务代码传如的value是否包含权限列表的某一项，包含就返回
                return (
                    all_permission === permission ||
                    permissionFlag.includes(permission)
                );
            });
            // 返回false说明没有权限，当前指令绑定的el.parentNode.removeChild(el)
            if (!hasPermissions) {
                el.parentNode && el.parentNode.removeChild(el);
            }
        } else {
            throw new Error(`请设置操作权限标签值`);
        }
    },
};

大文件上传下载 ​

文件转为二进制流，利用流可切分性（Blob.slice()）把大文件切分上传，等文件都发送完毕，在发送合并的网络请求/后端直接判断是否发完返回状态码

参考：https://juejin.cn/post/6844904046436843527https://blog.csdn.net/tck001221/article/details/121626930

思路：

  1. 拿到file对象

  2. 切片Blob.slice()

    • 断点： 传文件时，数组保存正在传的请求实例，若请求成功，就把该实例从数组中删除，点取消，遍历数组，依次调用abort方法
    • 续传：传输切片前，把该文件生成的hash值传给服务器，若没上传成功，但上传部分切片，就把上传的切片名称返回，前端拿到切片做个过滤，把没匹配成功的重新匹配
    • 秒传：传切片前，把该文件生成的hash值传给服务器，验证是否传输过，若已传过，就直接返回传输成功
    • 重传 ： 之前有失败的切片可以记录chunkid，只传失败的切片

  3. 合适的切片大小 2M/10M：10 * 1024 * 1024 Blob.prototype.slice返回原文件的某个切片也就是file.slice(cur, cur + size)动态根据文件大小切分-文件大小是知道的，带宽用常规带宽）文件大小/带宽 = 每个分片大小 ，向上取整 就出来了1M宽带即指1Mbps=1000Kbps=1000/8KBps=125KBps根据文件大小，文件大小小于等于10MB的按照32KB，10MB~100MB按照1MB,超过100MB按照10MB进行分片。

    http 可并发性，同时上传多个切片, 传输到服务端的顺序可能会发生变化，需要给每个切片记录顺序请求并发控制async-pool插件

  4. 切片数组：chunkList[{ chunk: fileObj.slice(curChunk, curChunk + chunkSize),    fileName: ${hash}_${i}.${suffix} // 文件名规则按照 hash_1.jpg 命名}]

  5. server端

    • 何时合并切片：切片什么时候传输完成  每个切片中携带切片最大数量，服务端接受到这个数量的切片时自动合并。额外发一个请求，主动通知服务端进行切片的合并
    • 如何合并切片  Nodejs读写流（readStream/writeStream），将所有切片的流传输到最终文件的流里

  6. hash优化：webworker/抽样hash, 或者 切片依次传入spark-md5中，等切片传完后，用postMessage把生成的hash传给主线程

js
// 将文件按固定大小（2M）进行切片
const chunkSize = 2097152,
	chunkList = [], // 保存所有切片的数组
	chunkListLength = Math.ceil(fileObj.size / chunkSize), // 计算总共多个切片
	suffix = /\.([0-9A-z]+)$/.exec(fileObj.name)[1] // 文件后缀名
// 根据文件内容生成 hash 值
const spark = new SparkMD5.ArrayBuffer()
spark.append(buffer)
const hash = spark.end()
// 生成切片，这里后端要求传递的参数为字节数据块（chunk）和每个数据块的文件名（fileName）
let curChunk = 0 // 切片时的初始位置
for (let i = 0; i < chunkListLength; i++) {
	const item = {
		chunk: fileObj.slice(curChunk, curChunk + chunkSize),
		fileName: `${hash}_${i}.${suffix}` // 文件名规则按照 hash_1.jpg 命名
	}
	curChunk += chunkSize
	chunkList.push(item)
}

websocket的封装 ​

http请求应答，wesocket可以主动推，客户端监听事件，发布订阅模式

  1. mixin混入，存到vuex
  2. 初始化new WebSocket('ws://localhost:9999');发布订阅模式 请求连接
  3. server收到连接请求返回101，客户端ws.onopen
  4. 客户端ws.onmessage收到数据，重置心跳检测开启timer1 ， 每隔timer2发一次，ws.send("ping")，只要server有返回就重置心跳定时器
  5. 当连接失败/断网，客户端ws.onerror或者timer3时间内没收到 触发ws.onclose，客户端 timer时间后重连初始化

说明

  1. 连接成功/收到数据，重置心跳检测
  2. 心跳检测的过程中，一段时间没收到数据/连接失败/网络断开 ， 重连
  3. timer1 防止重连次数频繁，隔10s请求一次，记录发重连间隔的定时器
  4. timer2  每隔30s发一次心跳，记录发ping间隔的定时器
  5. timer3 记录 timer3时间内没收到再重连，这里注意，一段时间内没收到说明服务端主动断开，不要直接重连，先close 再 重连，close会触发重连

心跳机制和断线重连 刷新页面，断网，长时间不通信会断开

前端定时发心跳消息 ping，后端收到后立刻返回 pong ，告知连接正常，一定时间没有 pong ，前端重连

  1. 重连操作 reconnect： 网络断开，执行 reconnect，使用一个定时器，4秒重新创建新 websocket ，调 createWebSocket 函数，重新会执行及发送数据给服务器端。

var lockReconnect = false;//避免重复连接
function reconnect(url) {
  if(lockReconnect) {
    return;
  };
  lockReconnect = true;
  //没连接上会一直重连，设置延迟避免请求过多
  tt && clearTimeout(tt);
  tt = setTimeout(function () {
    createWebSocket(url);
    lockReconnect = false;
  }, 4000);
}
